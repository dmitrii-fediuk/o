#
Я разрабатываю `᛭L`: свой полуформальный язык для описания онтологий (`᛭O`) для LLM.

#
`S` ≔  
```
Утверждение, которые обладает следующими свойствами (некоторыми или всеми):
- Его истинность неочевидна.
- Его истинность нуждается в проверке.
- Его истинность не должна быть принята LLM на веру.
LLM должен относиться к ним осторожно: как к предположениям.
```

#
`S` — это далеко не всегда утверждения LLM.
Обычно это: 
##
Утверждения моих клиентов в рамках задач, которые они хотят решить/
##
Мои собственные предположения из любой предметной области, которые я хочу проанализировать.
Например:
```
`H1?` ≔? (`GEᨀ` может быть более эффективным для меня способом решения `T⁎`, чем `Gᨀ`)
```

# 
Для анализа `S` я обычно даю LLM запрос следущей структуры (здесь я обозначаю `S` посредством `H1?`: как и в моём примере выше):
~~~markdown

## 1.1.
`H`[§1-§4] ≔ `H1?`

## 1.2.
`Ps`[§1-§4] ≔ ⠿~(доводы за `H`)

## 1.3.
`Pi`[§1-§4] : `Ps`

## 1.4.
`Cs`[§1-§4] ≔ ⠿~(доводы против `H`)

## 1.5.
`Ci`[§1-§4] : `Cs`

## 1.6.
`As`[§1-§4] ≔ (`Ps` ⋃ `Cs`)

 ## 1.7.
`Ai`[§1-§4] : `As`

## 1.8.
`AiS`[§1-§4] ≔ («Score» для `Ai`: целое число от 0 до 100, отражающее значимость `Ai`)

## 1.9.

# 2. `᛭T`
Действуй последовательно, по следующим шагам:
2.1. Определи и систематизируй `Cs`.
2.2. Определи и систематизируй `Ps`.
2.3. Совместно проанализируй `Cs` и `Ps`. 
Для каждого `Ai` определи `AiS`. 
2.4. На основе этого анализа составь `V` и определи `HS`.
2.5. Предоставь `R` в соответствии с требованиями §3
~~~

# Markdown / Заголовки
## Структура твоего ответа
### 
У твоего ответа не должно быть одного общего заголовка 1-го уровня (`#`), потому что твой ответ будет вставлен внутрь секции 1-го уровня (`#`) другого документа Markdown.
###
Исходя из предыдущего пункта, в качестве заголовков верхего уровня ты должен использовать заголовки 2-го уровня (`##`).
Таких заголовков должно быть несколько: тем самым ты разбиваешь свой ответ на разделы.
Если твой ответ краток, то не разбивай его на разделы вообще.
### 
Разумеется, ты также можешь использовать заголовки более нижних уровней внутри заголовков 2-го уровня: для дополнительной структуризации текста.

## 
Никогда не используй выделение жирным (`**`) в заголовках.

## 
Всегда форматируй заголовки только символами решётки (`#`), не другими способами. 

## 
Не вставляй самовольно полностью пустые строки после заголовков Markdown.

#### Неправильное форматирование
```markdown

```
#### Правильное форматирование
```markdown

```
