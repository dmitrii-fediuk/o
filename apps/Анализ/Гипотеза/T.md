# 1.
⌠`᛭Dс`⌡ ⟨ 
	`H` ≔ ꖿ STUB
	`P⠿` ≔ ⠿~ ⟨ доводы за `H` ⟩
	`Pᵢ` ᗉ `P⠿`
	`C⠿` ≔ ⠿~ ⟨ доводы против `H` ⟩
	`Cᵢ` ᗉ `C⠿`
	`A⠿` ≔ `P⠿` ⋃ `C⠿`
	`Aᵢ` ᗉ `A⠿`
	`S⧙Aᵢ⧘` ≔ ⟨ «Score» для `Aᵢ`: целое число от 0 до 100, отражающее значимость `Aᵢ` ⟩
	`V` ≔ ⟨ твоё итоговое, обоснованное, объективное мнение о `H` ⟩
	`SH` ≔ ⟨ «Score» для `H`: целое число от 0 до 100, отражающее твою оценку `H` ⟩
	`R` ≔ ⟨ итоговый отчёт ⟩
	`T⠿` ≔ `A⠿` ⋃ ⠿{`V`}
	`Tᵢ` ᗉ `T⠿`
⟩

# 2. `᛭T`
Действуй последовательно, по следующим шагам:
##
Определи и систематизируй `C⠿`.
##
Определи и систематизируй `P⠿`.
##
Совместно проанализируй `A⠿`. 
##
Для каждого `Aᵢ` определи `S⧙Aᵢ⧘`. 
##
На основе этого анализа составь `V` и определи `SH`.
##
Предоставь `R` в соответствии с требованиями §3.

# 3. Требования к формату `R`
##
`Rs` ≔
```markdown
## 1. Pros
### 1.1. <`P₁` Title>
**Score**: **`S⧛P₁⧚`**
`P₁`

### 1.2. <`P₂` Title>
**Score**: **`S⧛P₂⧚`**
`P₂`

<…>

## 2. Cons
### 2.1. <`C₁` Title>
**Score**: **`S⧛C₁⧚`**
`C₁`

### 2.2. <`C₂` Title>
**Score**: **`S⧛C₂⧚`**
`C₂`

<…>

## 3. Verdict
**Score**: **`SH`**
`V`
```

##  
`R` должен быть документом Markdown структуры `Rs`.

##
Не заключай `R` внутрь backticks.  
Другими словами, в твоём ответе `R` должен отображаться как rendered Markdown, а не как сырая разметка Markdown.

## 
###
Каждый абзац `Tᵢ` должен содержать ровно одно предложение.
###
Между абзацами `Tᵢ` не должно оставаться пустых строк.
